Making Of - Documentação do Projeto

1. Modelação de Dados

O processo de modelação começou no papel, onde foram definidas as entidades centrais (Pessoa, Projeto, Tecnologia, UC). A relação Many-to-Many entre Projetos e Tecnologias foi crucial para permitir a reutilização de tags de stack tecnológica.

Figura 1: Esboço inicial do Diagrama Entidade-Relação.

2. Implementação de Formulários CRUD

Nesta fase foram implementadas operações CRUD (Create, Read, Update, Delete) para as entidades Projeto, Tecnologia, Competência e Formação. O Django facilita muito este processo através das ModelForm, que geram automaticamente os campos do formulário a partir do modelo.

Para cada entidade foram criadas três views: uma para criar, uma para editar (que recebe o id do objeto via URL) e uma para apagar. Todas partilham o mesmo template form_generico.html, recebendo apenas o título como contexto diferente — o que demonstra bem a reutilização de código no Django.

Dificuldades encontradas

  • Nomes de URL inconsistentes: Os templates usavam nomes como novo_projeto mas o urls.py definia projeto_novo. Isto causava erros NoReverseMatch em tempo de execução. A solução foi uniformizar todos os nomes no urls.py.
  • Template inexistente (layout.html): Alguns templates faziam {% verbatim %}{% extends 'portfolio/layout.html' %}{% endverbatim %} quando o ficheiro base se chama base.html. Resultou no erro TemplateDoesNotExist nas páginas de Projetos e Tecnologias.
  • Rotas em falta: As páginas /portfolio/competencias/ e /portfolio/formacoes/ não tinham rotas definidas no urls.py, resultando em erro 404.
  • Templates vazios: Os ficheiros competencias.html e formacoes.html existiam na pasta mas estavam completamente vazios.

O que aprendi e considerei importante

  • O Django é muito vantajoso na construção de aplicações web porque o padrão MTV (Model-Template-View) separa bem as responsabilidades, tornando o código mais organizado e fácil de manter.
  • As ModelForm poupam imenso tempo — não é necessário escrever HTML de formulário manualmente, o Django gera os campos automaticamente com validação incluída.
  • O sistema de namespaces nos URLs (ex: portfolio:novo_projeto) é muito útil para evitar conflitos entre apps e manter os links independentes dos URLs reais.
  • Os botões de editar/apagar só aparecem para utilizadores autenticados com o grupo gestor-portfolio, o que é uma boa prática de segurança mesmo sem middleware adicional.

3. Dificuldades Técnicas Gerais

  • Importação de Dados: O mapeamento dos campos da API da Lusófona para os campos do modelo Django exigiu ajustes no script para tratar caracteres especiais.
  • Roteamento: Corrigido o erro de 404 ao aceder ao URL sem a barra final através da configuração correta no urls.py do projeto.
Metodologia

Implementação incremental: primeiro os modelos, seguidos do admin para teste de dados, e finalmente as views, formulários e templates.
